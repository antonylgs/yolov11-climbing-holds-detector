# Learning notes — Issue 003: Dataset preparation

Roboflow exports in → one clean, single-class, deterministically-split dataset out. This is the unglamorous plumbing that decides whether training (issues 004–005) works or silently lies to you.

## What this issue accomplished

```
src/holds_detector/dataset.py        all the logic (discover → validate → split → write)
scripts/prepare_dataset.py           thin CLI entrypoint
scripts/generate_dataset_fixture.py  builds the fake Roboflow exports in tests/fixtures/dataset/
tests/test_dataset.py                10 tests, incl. a real 1-epoch training smoke
```

Try it (on the fixture — real exports drop in the same way):
```bash
uv run python scripts/prepare_dataset.py \
    tests/fixtures/dataset/export_a tests/fixtures/dataset/export_b
# → data/dataset/{train,val,test}/{images,labels}/ + configs/dataset.yaml
```

## The big ideas, one by one

### 1. Why a script at all? Datasets are code's ugly sibling
Real datasets are messy: multiple sources with different class lists (`['hold','volume']` vs `['climbing-hold']`), missing labels, annotation-tool glitches producing 2-point "polygons". If you hand-assemble the dataset once in Finder, you can never reproduce it, and every retrain (issues 005, 008) starts from an unknown state. The script makes the dataset a *build artifact*: delete `data/dataset/`, rerun, get the exact same thing. That's LESSON.md §12 step 7 ("iterate: add/fix data → retrain") made cheap.

### 2. Split by image, never by annotation
The unit of splitting is the *photo*, not the hold. If two holds from the same photo landed in train and val, the model would be validated on pixels it already saw — overfitting's smoke alarm (LESSON.md §6) disabled exactly when you need it. The code never even represents per-annotation granularity in the split; it's structurally impossible to leak.

### 3. Determinism is a feature you build, not hope for
"Seeded shuffle" alone isn't enough — `rglob` order depends on the filesystem. The recipe: **sort by a stable key first, then shuffle with `random.Random(seed)`**. Same inputs + seed = byte-identical output, on any machine, in any order the OS lists files. The test proves it by SHA-256-hashing every output file across two runs. Why care? When model v2 beats v1, you must be able to say "same data, better training" or "same training, better data" — a non-reproducible split makes that comparison meaningless.

### 4. Validate before you train, because YOLO won't
Ultralytics happily trains around bad labels (skips, warns into a wall of logs, or worse, learns from garbage). The validation pass catches each defect type explicitly: missing/empty label files, lines with < 3 points, zero-area polygons (collinear points — yes, annotation tools produce these), out-of-bounds coordinates. Two modes:
- **default = quarantine**: defective pairs are copied to `data/dataset/quarantine/` for eyeballing, excluded from splits, listed in `validation_report.json`. The pipeline keeps moving.
- **`--strict` = fail**: nothing is written, exit 1. For CI or "I just labeled 100 photos, did I mess any up?"

One finding quarantines the *whole image* — a half-trusted label file isn't worth training on.

### 5. Class remapping: where LESSON.md §5 becomes one line of code
Every label line's class id is replaced with `0`, whatever the source called it. Subtlety worth stealing: coordinates are passed through as **verbatim text tokens**, never parsed to float and re-formatted. No precision drift, and byte-identical output stays trivially true.

### 6. Fixture-driven testing — fake the *shape*, not the *content*
The test fixtures are three miniature Roboflow exports generated by a seeded script: 96×96 synthetic walls with colored blobs, where labels are written *from the exact polygons drawn*. So the fixture is simultaneously: format-correct (tests the parser), defect-complete (one image per defect type, labels hand-written to be broken in exactly one way), and *trainable* (the final test runs a real 1-epoch fine-tune through the emitted `dataset.yaml`). The whole suite needs zero downloads. Same philosophy as issue 002's stubbed pipeline: control the inputs precisely, then test the real machinery.

### 7. Name collisions: the bug you'd hit in month two
`export_a/train/wall_0.jpg` and `export_a/valid/wall_0.jpg` are different photos with the same stem. Flatten naively and one silently overwrites the other — dataset shrinks, nobody notices, mAP wobbles mysteriously. Output names embed source index + split path (`00_export_a_train_wall_0`), and a test pins it. Generalizable lesson: whenever you merge trees into a flat namespace, *prove* uniqueness, don't assume it.

## Questions to test yourself

1. Why sort before the seeded shuffle? What breaks if you shuffle `rglob` output directly?
2. A single photo has 30 holds. Why must all 30 land in the same split, and which design choice makes the alternative impossible?
3. Why does the label rewriter keep coordinate tokens as text instead of parsing floats?
4. You add a second export also named `export_a` (different directory, same name). Will outputs collide? (Check how `source_id` is built.)
5. When would you run with `--strict` rather than the quarantine default?
